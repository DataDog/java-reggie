# Design: fix `HTML_TAG.matches()` giving up on the first retry candidate instead of continuing the search

Date: 2026-07-06
Status: **implemented and verified** (superseded the originally-scoped one-line fix with a full
O(1) rewrite — see §4a). Grounded in main repo, branch `perf/eliminate-slow-paths`.

## 1. Problem statement

While investigating a ~15% `matches()` slowdown vs. JDK on an adversarial `SPECIALIZED_BACKREFERENCE`
(HTML-tag-close) benchmark case, the original plan was to generalize `PINNED_BACKREFERENCE`'s
separator check to cover this shape and eliminate its retry loop. That plan does not work: the
retry loop in `generateHTMLTagFindFromMethod`/`generateHTMLTagMatchesMethod` is not there because
of *group-boundary* ambiguity (which a disjointness proof genuinely resolves — see
`doc/2026-07-06-backreference-pinned-boundary-design.md`) — it is there because the pattern's `.*`
segment can match arbitrary text containing false `"</"` occurrences that fail the backreference
check, and finding the *correct* occurrence requires searching past those false positives.
Disjointness between the captured group's charset and the delimiter does not eliminate that need.

Investigating further (reading the actual generated bytecode, then confirming with a live run,
not just the doc's pseudocode) surfaced a genuine, pre-existing correctness bug instead, unrelated
to performance:

```
pattern: <(\w+)>.*</\1>
input:   <a>x</a>y</a>
JDK        input.matches(pattern)              => true
Reggie     Reggie.compile(pattern).matches(input) => false   (WRONG)
```

Confirmed directly (not inferred) via a throwaway JUnit case run through
`./gradlew :reggie-runtime:test`, using the actual compiled `RuntimeCompiler` output — not a
hand-traced reading of the bytecode.

## 2. Root cause

`BackreferenceBytecodeGenerator.generateHTMLTagMatchesMethod`
(`reggie-codegen/src/main/java/com/datadoghq/reggie/codegen/codegen/BackreferenceBytecodeGenerator.java:264-454`)
implements `matches()` for the `HTML_TAG` pattern shape (`<(\w+)>.*</\1>`) as: capture the tag name
via a `\w+` scan, then loop `input.indexOf("</", pos)` looking for a candidate close tag, checking
at each candidate whether (a) the backreference matches the captured tag name and (b) a `'>'`
immediately follows. When both hold, the method assumes it has found *the* match and — at
line 434 — checks whether that occurrence's end position equals `len` (`matches()` requires full
input consumption):

```java
// line 425-437
mv.visitVarInsn(ILOAD, backrefPosVar);
mv.visitVarInsn(ILOAD, group1LenVar);
mv.visitInsn(IADD);
mv.visitInsn(ICONST_1);
mv.visitInsn(IADD);
mv.visitVarInsn(ILOAD, lenVar);
Label returnTrue = new Label();
mv.visitJumpInsn(IF_ICMPNE, returnFalse);   // <-- BUG: jumps to returnFalse, not tryNext
mv.visitLabel(returnTrue);
mv.visitInsn(ICONST_1);
mv.visitInsn(IRETURN);

// Try next position
mv.visitLabel(tryNext);
mv.visitVarInsn(ILOAD, closeStartVar);
mv.visitInsn(ICONST_1);
mv.visitInsn(IADD);
mv.visitVarInsn(ISTORE, posVar);
mv.visitJumpInsn(GOTO, searchLoop);
```

If the length check fails — the found `</tagname>` occurrence has the right tag name and a
trailing `'>'`, but leaves trailing content in the input — the method jumps to `returnFalse` and
gives up immediately, instead of jumping to `tryNext` (which advances `pos` past this candidate
and resumes the `indexOf` search for a *later* occurrence of the same tag name that *would* reach
end-of-string). In `<a>x</a>y</a>`, the first `</a>` (at index 4) has the right name and a
trailing `>`, but the input continues with `y</a>` afterward — so the correct behavior is to keep
searching and find the second `</a>` (which does reach end-of-string), not to bail out.

Contrast with `generateHTMLTagFindFromMethod`
(`BackreferenceBytecodeGenerator.java:459-655`), which has no end-of-string constraint (`find()`
only needs *a* valid completion to exist, not a specific span) and therefore returns on the first
successful backreference match unconditionally — that is correct for its own semantics, and is
not the buggy pattern. The bug is specific to the length-check branch inside
`generateHTMLTagMatchesMethod`.

## 3. Blast radius

- **`matches(String)`** (`HTML_TAG` shape only): buggy, as shown above.
- **`match(String)`** (`BackreferenceBytecodeGenerator.java:2071-2100`): calls `matches()` first
  and returns `null` immediately if it's `false` — inherits the bug transitively. Confirmed by
  reading the generated call sequence: `matches()` → `IFNE matched` → `ACONST_NULL`/`ARETURN` on
  the false branch.
- **`matchesBounded`/`matchBounded`** (`BackreferenceBytecodeGenerator.java:2113-2181`): both
  delegate to `matches(String)` (after extracting the bounded substring) — inherit the bug too.
- **NOT affected**: `find()`/`findFrom()` (no end-of-string constraint, as above) and
  `findMatch()`/`findMatchFrom()`/`findMatchBounded` — these delegate to
  `BackreferenceHelpers.findHtmlTagMatchFrom` (`generateHtmlTagFindMatchFromMethod`,
  `BackreferenceBytecodeGenerator.java:2266-2286`), an independent code path from
  `generateHTMLTagMatchesMethod`, with no end-of-string requirement to mishandle.
- **NOT affected**: `ATTRIBUTE_MATCH` (`"([^"]+)"\s*=\s*"\1"`) — its content scan is bounded by a
  literal `"` with no `.*`-style wildcard in between, so there is no possibility of a false
  delimiter occurrence and no retry loop to begin with; structurally cannot exhibit this bug.
- **NOT affected**: `GREEDY_ANY_BACKREF` (`(.*)\d+\1`) — its `matches()`
  (`generateGreedyAnyBackrefMatchesMethod`, `BackreferenceBytecodeGenerator.java:1583+`) is
  genuine exhaustive longest-first backtracking over capture length, not a "stop at the first
  successful occurrence" shortcut, so it does not have this failure shape (out of scope to verify
  further here — flagged only for completeness, not part of this fix).
- **Not currently caught by the differential fuzzer's tracked budget**: grepped
  `doc/fuzz/2026-06-29.md` (the current 28-finding budget document) for any HTML-tag-shaped
  repro; none found. This appears to be a new, previously-undiscovered divergence, not one already
  priced into `KNOWN_FINDINGS_BUDGET`. The property/random fuzzer's pattern grammar most likely
  doesn't generate literal `<`/`>`/`/`-shaped patterns often enough to have hit this specific
  repeated-tag-name shape; it's why a manual/targeted case found it and the fuzzer sweep didn't.

## 4. Proposed fix

Single, minimal, mechanical change to `generateHTMLTagMatchesMethod`
(`BackreferenceBytecodeGenerator.java:264-454`): change the length-mismatch branch's jump target
from `returnFalse` to `tryNext`, exactly mirroring how `generateHTMLTagFindFromMethod`'s equivalent
loop already handles a non-terminal candidate — the *only* difference between "keep searching" and
"give up" in this method already exists as the `tryNext` label (used by the backreference-mismatch
and trailing-`>` mismatch branches); the length check is simply wired to the wrong target.

```java
// Found valid closing tag! For matches(), check if at end
// return (backrefPos + group1Len + 1) == len;
mv.visitVarInsn(ILOAD, backrefPosVar);
mv.visitVarInsn(ILOAD, group1LenVar);
mv.visitInsn(IADD);
mv.visitInsn(ICONST_1);
mv.visitInsn(IADD);
mv.visitVarInsn(ILOAD, lenVar);
Label returnTrue = new Label();
mv.visitJumpInsn(IF_ICMPNE, tryNext);   // was: returnFalse
mv.visitLabel(returnTrue);
mv.visitInsn(ICONST_1);
mv.visitInsn(IRETURN);
```

`tryNext` is already defined later in the same method (currently at line 440) and already performs
exactly the needed `pos = closeStart + 1; goto searchLoop;` continuation — no new labels, no new
local variables, no change to the loop's exit condition (`searchEnd`, reached when `indexOf`
returns `-1`, still correctly falls through to `returnFalse`). This is a one-line change (the jump
target on an existing `IF_ICMPNE`), not a restructuring.

No corresponding change is needed in `generateHTMLTagFindFromMethod` (not affected, per §3).

**Revised during implementation**: `BackreferenceHelpers.findHtmlTagMatchFrom` turned out *not* to
be unaffected, contradicting §3's original claim — see §4a.

## 4a. What was actually implemented (supersedes §4)

Two things changed beyond the original one-line-fix plan, both discovered during implementation
and verification, not anticipated in the original design:

**A second, related bug in `findHtmlTagMatchFrom`.** After applying the §4 fix, the new regression
test's `match()` assertion failed: `matches("<a>x</a>y</a>")` now correctly returned `true`, but
`match("<a>x</a>y</a>").end()` returned `8`, not `13`. Root cause: `match(String)` calls `matches()`
to gate, then delegates to `findMatchFrom` for the span — but
`BackreferenceHelpers.findHtmlTagMatchFrom` (`reggie-runtime/src/main/java/com/datadoghq/reggie/runtime/BackreferenceHelpers.java:42-81`)
independently returns on the *first* successful closing-tag occurrence, same shape of bug as §2 but
in the rich-API span-reporting path rather than the boolean path. Confirmed against real JDK
behavior (not assumed): `Pattern.compile("<(\\w+)>.*</\\1>").matcher("<a>x</a>y</a>").find()`
reports `[0,13)`, the *longest* valid completion, matching greedy `.*` semantics. Fixed by
continuing the search past a successful candidate and returning the *last* one found for a given
start position, rather than the first (`BackreferenceHelpers.java:61-90` in the current code).

**The retry loop in `generateHTMLTagMatchesMethod` was replaced entirely, not just fixed.**
Profiling the post-fix ~15% `matches()` throughput gap vs. JDK (async-profiler, confirmed
`indexOf`/`ArraysSupport.mismatch` as the hot path, see the profiling task's findings) led to a
sharper insight than "keep the retry loop but stop giving up early": `matches()` requires the whole
input consumed, so once `group1Len` is known from the initial `\w+` scan, the pattern's total
length is fixed — `1 + group1Len + 1 + k + 2 + group1Len + 1 == len` for gap length `k >= 0` —
which pins the closing tag's start position to exactly one value: `closeIdx = len - group1Len - 3`.
No search is needed in either direction; a single computed-position check suffices. Verified the
formula by direct calculation against `<a>x</a>y</a>` (closeIdx=9), `<a>p</a>q</a>r</a>`
(closeIdx=14), and the exact benchmark fixture shape before touching code. The entire
`indexOf`-loop-with-retry block in `generateHTMLTagMatchesMethod` was replaced with: compute
`closeIdx`, verify `closeIdx >= pos`, then four direct checks (`'<'`, `'/'`, `regionMatches`
against the captured group, trailing `'>'`) — no loop, no labels beyond the existing
`returnFalse`. Current implementation: `BackreferenceBytecodeGenerator.java`
(`generateHTMLTagMatchesMethod`, ~lines 264-440 post-rewrite).

This generalizes beyond HTML/XML syntax specifically: the same closed-form trick applies to any
pattern of the shape `FixedPrefix + Group(\N) + FixedInfix + UnconstrainedGap(.*) + FixedDelimiter
+ \N + FixedSuffix` matched via `matches()`, where every piece except the `.*` gap has a
statically-known length. As shipped, it only benefits the literal `<`/`>`/`</`/`>` shape because
`detectHTMLTagPattern` (the routing detector) only recognizes that exact literal-character
structure — generalizing the *detector* to any fixed-delimiter/single-gap/single-backref shape is
explicitly out of scope here (§6).

This does **not** apply to `find()`/`findFrom()`/`findMatchFrom()`/`findMatch()` — those don't
require full-input consumption, so the gap length isn't algebraically pinned down; a genuine
(now correctness-fixed, but not O(1)) search remains necessary there.

  covers the §1 repro (`matches()` now `true`), a three-closing-tag case

- `StrategyCorrectnessMetaTest -Dreggie.metatest.enforce=true`: 4/4 pass.
- Differential fuzzer gate (`AlgorithmicFuzzTest.divergenceGate_enforcedViaProperty`): 28 findings,
  exactly at the existing `KNOWN_FINDINGS_BUDGET`; every repro cross-checked against the E1–E6
  catalogue in `doc/fuzz/2026-06-29.md` — all pre-existing, none HTML-tag-shaped. Zero new
  divergences; no budget re-baseline needed.
- Targeted scratch stress-testing (written, verified, deleted per session convention): empty tag
  name rejection, minimum-length boundary (`<a></a>`), tag name as a substring of another word
  (`<abc>x</ab>` correctly rejected), multiple candidate closing occurrences before *and* after the
  correct one, and a 5000-input randomized fuzz of `<(\w+)>.*</\1>` compared directly against
  `String.matches()` (JDK) — **0 mismatches**.
- Full `./gradlew spotlessApply` + `./gradlew build` (including `check`/`jacocoVerify`): green.
- JMH re-measurement of the exact benchmark that originally showed the ~15% gap (2 forks × 5
  warmup + 5 measurement iterations, numbers pulled directly from the raw
  `reggie-benchmark/build/reports/jmh/results.json`, not just agent-summarized):

  | Benchmark | Before | After |
  |---|---|---|
  | `reggieHtmlTagAdversarialMatch` | 936.2 ops/ms | **158,028.3 ± 33,890.8 ops/ms** |
  | `jdkHtmlTagAdversarialMatch` (control, untouched) | 1,074.8 ops/ms | 1,072.0 ± 37.9 ops/ms |
  | `reggieHtmlTagAdversarialNoMatch` | 964.5 ops/ms | **430,168.1 ± 26,269.4 ops/ms** |
  | `jdkHtmlTagAdversarialNoMatch` (control, untouched) | 166.4 ops/ms | 152.8 ± 18.7 ops/ms |

  Reggie went from ~13–15% slower than JDK to ~147x faster on Match, and from ~5.8x faster to
  ~2,816x faster on NoMatch — the NoMatch jump is larger than Match's because the *old* algorithm's
  worst case was exhaustive-scan-to-failure, which the new O(1) check eliminates equally for both
  outcomes (verified by hand: for the NoMatch fixture, the computed `closeIdx` lands mid-word
  inside `"section>"`, failing the very first `charAt` check rather than scanning ~200 false
  candidates). JDK's own numbers are stable across both runs (unmodified code, normal JMH noise),
  confirming the measurement environment itself didn't shift.

## 6. Explicitly out of scope

- The originally-proposed `PINNED_BACKREFERENCE` separator generalization for the tag-close shape
  — abandoned; per §1, disjointness doesn't eliminate the need for the retry loop in this shape,
  so there is no valid "single forward pass, no retry" version of `<(\w+)>.*</\1>` to build via
  that route (the O(1) closed-form trick in §4a is a different, unrelated mechanism).
- Generalizing `detectHTMLTagPattern` (or adding a new detector) to recognize the broader
  `FixedPrefix + Group + FixedInfix + .* + FixedDelimiter + \N + FixedSuffix` shape for arbitrary
  (non-`<`/`>`) delimiters — the algebraic trick in §4a would apply, but nothing routes such a
  pattern to this generator today; a real follow-up if this shape shows up with other delimiters.
- Applying the same O(1) closed-form technique to `find()`/`findFrom()`/`findMatchFrom()`/
  `findMatch()` — not possible as-is, since those don't require full-input consumption (see §4a);
  a *different* optimization (searching backward from the end, mirroring JDK's own greedy-backtrack
  direction, instead of forward from the start) could still help there but wasn't pursued.
- Auditing `GREEDY_ANY_BACKREF`'s or `ATTRIBUTE_MATCH`'s `matches()` methods further than the
  scoping already done in §3 — flagged as apparently not affected, not exhaustively re-verified
  beyond that structural argument.
- Any change to `PatternAnalyzer`'s routing, `FallbackPatternDetector`'s guards, or
  `PinnedBackreferenceInfo`/`PinnedBackreferenceBytecodeGenerator` — this fix is entirely contained
  within `BackreferenceBytecodeGenerator`'s/`BackreferenceHelpers`' existing `HTML_TAG` codegen and
  touches nothing from this session's `PINNED_BACKREFERENCE` work.
