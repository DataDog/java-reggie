# Reggie 1.0.0 public-release readiness — assessment (2026-06-18)

Re-evaluation after the 2026-06-11→18 capture-correctness + performance work. Evidence-backed
(file:line / gate output / build). Current version: **`0.4.0-SNAPSHOT`** (build.gradle:8).

## Verdict: **NOT clean for a public 1.0.0 yet — no catastrophic blocker, but a clear pre-1.0 punch list.**

The engine is in strong shape — build green, ReDoS-safe, boolean (accept/reject) correctness solid,
and throw-by-default protects callers from silent wrong answers. But several items must be closed (or
explicitly scoped/documented) before publishing a "drop-in, faster `java.util.regex`" 1.0. Recommend
shipping a **0.5.0 / 1.0.0-RC** first and gating 1.0.0 on the punch list below.

## Pre-1.0.0 punch list (recommended gating)

**P0 — correctness narrative & measurement (must do before any public correctness claim):**
1. Reconcile the AGENTS.md / README "correctness guarantee" prose with the actual **throw-by-default**
   behavior. State plainly: native-or-throw by default; opt-in JDK fallback.
2. Establish a single source of truth for the **PCRE conformance** number + denominator; confirm the
   conformance/integration suite measures **Reggie** (not JDK) — adversarial Finding 2.
3. State the fuzz guarantee honestly: "zero *boolean* divergences; N known native group-span gaps on
   degenerate inputs, all O(n)/ReDoS-safe" — not "zero divergences."

**P1 — robustness:**
4. Confirm or document **thread-safety** of cached matchers (Finding 6) — a "drop-in `java.util.regex`"
   claim implies safe concurrent reuse; `java.util.regex.Pattern` is thread-safe, `Matcher` is not.
   Define and document Reggie's contract; add a concurrency stress test.
5. Make malformed-pattern parsing throw a typed `PatternSyntaxException` (Finding 8).

**P2 — capture-correctness gaps (can ship 1.0 with these documented as known limitations):**
6. Drive the 69-finding fuzz gate down (the deferred Class B empty-iteration + Class C DFA give-back),
   or **document** the residual as known, native, O(n), span-only, degenerate-input limitations and
   give the gate headroom (budget > exact count) so it isn't brittle at-budget.
7. Quantify real-world frequency of the residual span gaps (currently only fuzzer-shape evidence) to
   justify the "rare/degenerate" framing in the limitations doc.

**P3 — release hygiene:**
8. Add 0.4.0 (and the 1.0 entry) to CHANGELOG; add a 1.0 checklist.
9. Reconcile RELEASING.md vs AGENTS.md (SSM param names, published artifact coordinates); verify
   Maven Central creds are provisioned.
10. Resolve the two known perf regressions from benchmark-report.md (`(fo|foo)` group `match()` 0.57×;
    `^(a)(b)(c)$` ONEPASS_NFA `find()` 0.28×) or accept/document them.

## What this week materially improved toward 1.0
- The **IAST drain** is now O(n) and beats re2j 10–31× on adversarial inputs (the re2j-replacement
  goal) — see the week research report.
- **Throw-by-default** fallback policy + `@RegexPattern` delegating stubs make native coverage
  auditable (no silent JDK fallback).
- The **capture-correctness wave** closed the boolean silent-wrong-answer class and reduced the
  fuzz budget 50→18 (then the stronger `findAll` oracle re-exposed pre-existing span gaps as the 69).

output (**69**) and the green `./gradlew test` build. Re-validate any Finding before treating it as
current (e.g. Finding 6 thread-safety, Finding 2 suite-validity) — they are hypotheses to confirm.
